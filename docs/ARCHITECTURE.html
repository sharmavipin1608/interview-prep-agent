Architecture Reference — Multi-Agent Orchestration System

1. High-Level Overview

ClaudeTemplate is a multi-agent orchestration framework built on top of Claude Code. A single orchestrator reads tasks, routes them through a typed agent pipeline, and uses a three-tier memory system to maintain context across tasks and sessions.

2. Agent Pipeline

Every task flows through a typed agent pipeline. The orchestrator dispatches each agent with only the context it needs — no full conversation history. Two variants exist, selected by the classifier before each task.

Classifier — hooks/classify_task.sh

Runs as a PreToolUse hook before orchestrator dispatch. Writes verdict to /tmp/task_mode. Never reads file contents — only checks paths and git status. Caches result by task hash to avoid re-running on every tool call within the same task.

FORCE_FULL triggers

File path patterns

auth|jwt|session|password|secret|token|oauth

payment|billing|stripe|invoice|pricing

migration|schema.|flyway|liquibase

Dockerfile|docker-compose|.github/workflows

hooks/   CLAUDE.md   AGENTS.md

Structural signals

  • Any file deleted (D in git status)
  • Any new file created (?? in git status)
  • Modified file count > FAST_TRACK_FILE_LIMIT (default 5)

Dependency signals

New lines added to package.json, requirements.txt, go.mod, pom.xml, or major version bump in any manifest.

Sensitive keywords (in task description)

pii|gdpr|privacy|user.?data|email.?template|base\s+class

AMBIGUOUS → orchestrator decides

When no hard rule fires, the classifier writes AMBIGUOUS. The orchestrator then reasons inline (~200 tokens added to the existing orchestrator call — no separate LLM call).

Orchestrator question

Does this task introduce new behavior, touch shared logic, or carry risk not caught by pattern rules?

  • YES → full pipeline (log reason)
  • NO → fast-track (log reason)

3. Memory System

Three-tier Karpathy-style memory. Each tier has a distinct load strategy. Memory is pulled not pushed — agents grep for only what's relevant, never loading the full memory corpus.

Retrieval Strategy

Phase 1 — Now

Tag-based grep

grep "\[auth\]" memory/facts.md

Simple, zero infra. Works until ~100 fact entries.

Phase 2 — >100 facts

ChromaDB vector search

Embed facts at write time. Query by semantic similarity. Switch when grep results grow noisy.

Phase 3 — Long projects

Dedicated Memory Agent

Agent owns retrieval, curation, and consolidation of the memory corpus across all tiers.

facts.md Format

# One fact per line, prefixed with [domain] tag
[auth] JWT secret rotates every 24h
[database] PostgreSQL 15, schema in /db/schema.sql
[api] All responses wrapped in {data, error, meta}
[infra] Deployed on AWS ECS, us-east-1

4. Hooks

Hooks are shell scripts that Claude Code runs automatically at defined lifecycle events. Configured in .claude/settings.json. They run outside the LLM — zero token cost.

HookEventPurposeOutput
pre_task.sh PreToolUse Inject memory context once per session. Loads session_checkpoint.md and scratchpad.md into stderr for Claude to read. Deduplicates using session_id. Context written to stderr
classify_task.sh PreToolUse Bash classifier. Checks git status and task description against hard rules. Writes routing verdict. Caches by task content hash to avoid re-running. /tmp/task_mode
budget_guard.sh PreToolUse Counts tool calls in logs/tool_calls.log for today. Halts or warns if daily_call_limit exceeded. Configurable via CLAUDE_DAILY_CALL_LIMIT and CLAUDE_BUDGET_MODE env vars. Exit 1 to halt, or warning to stderr
log_tool.sh PreToolUse PostToolUse Appends every tool call to logs/tool_calls.log with timestamp, agent, and tool name. logs/tool_calls.log
post_task.sh PostToolUse Append to episodic log, update facts.md if a new decision was made, clear scratchpad. Memory files updated
on_error.sh Stop Log failure, requeue task in TASKS.md, send alert. Fail-safe — never crashes the pipeline. logs/ + TASKS.md update

→

pre_task.sh runs first so task context is available when classify_task.sh reads TASKS.md.

5. Agents — Detailed Breakdown

Each agent runs in isolation. The orchestrator passes only: task description + relevant memory chunks + the skill file for that agent. No full conversation history.

Researcher

Full pipeline only
Trigger
Unknown domain, new technology, need external context before coding
Input
Task description + memory/core.md
Output
Research findings written to memory/facts.md (tagged)
Skip in
Fast-track pipeline (domain already understood)
Skill
agents/researcher.md

Coder

Both pipelines
Trigger
Any implementation task — always follows TDD (write failing test, implement, pass)
Input
Task description + memory/scratchpad.md + skills/coding-patterns.md
Output
Code changes committed (not pushed), tests passing
Discipline
YAGNI — no extra features. DRY. No implementation before failing test.
Skill
agents/coder.md + skills/coding-patterns.md

Reviewer

Full pipeline only
Trigger
After Coder completes
Input
Code diff + skills/api-design.md
Output
PASS or fix list. If fix list → Coder re-runs.
Checks
Conventions (CONVENTIONS.md), API design patterns, correctness
Skip in
Fast-track pipeline (scope too small to warrant design review)

Tester

Both pipelines
Trigger
After Reviewer PASS (full pipeline) or after Coder (fast-track)
Input
Code + skills/test-strategy.md
Output
Integration and acceptance tests written and passing
Scope
Coder writes unit tests via TDD. Tester adds integration + acceptance layer.

Security — Hard Gate

Both pipelines · NEVER skippable
Trigger
After Tester — always, in every pipeline variant
Input
Code diff + skills/security-rules.md
Output
PASS → pipeline continues. BLOCKERS → pipeline stops entirely.
Checks
OWASP Top 10, injection, secrets in code, auth bypass, unvalidated input at boundaries
Rule
Golden Rule #6 — Security is a gate, never skip it

Git

Both pipelines · after Security PASS
Trigger
After Security PASS
Input
Code diff + skills/git-commit.md
Output
Atomic commit with conventional commit message, pushed to remote
Discipline
Never amends published commits. Frequent small commits. No --no-verify.

Memory

Both pipelines · NEVER skippable
Trigger
After Git — mandatory after every completed pipeline task. Also ad-hoc on significant decisions.
Input
Task output + memory/scratchpad.md + current memory/facts.md
Output
Updated facts.md, new session_checkpoint.md, appended episodic/YYYY-MM-DD.md, cleared scratchpad.md
Why mandatory
System coherence — without this, decisions are lost between sessions

Changelog (on-demand)

End of day / end of sprint
Trigger
End of day or end of sprint — not part of the per-task pipeline
Input
git log since last changelog entry
Output
CHANGELOG.md updated with human-readable summary of changes

Writer (on-demand)

Documentation tasks
Trigger
When documentation is explicitly needed — runs outside the main pipeline
Input
Task description + memory/core.md
Output
Markdown documentation files

6. Skills — Lazy Load

Skills are reference documents injected into the relevant agent's context. They are never preloaded — each agent receives only its own skill file.

FileLoaded byContains
skills/coding-patterns.md Coder agent TDD workflow, naming conventions, code structure patterns for this project's stack
skills/api-design.md Reviewer agent REST/API design standards, response envelope format, versioning rules
skills/test-strategy.md Tester agent Test pyramid, integration test patterns, what to cover at each level
skills/security-rules.md Security agent OWASP Top 10, project-specific security constraints, what to block vs. warn
skills/git-commit.md Git agent Conventional commits format, branch naming, PR conventions

8. Golden Rules

  • 1 Orchestrator stays thin — plan and delegate only, never write code or run tests
  • 2 Sub-agents get surgical context — no history, no fluff; only task + relevant memory + skill file
  • 3 Memory is pulled not pushed — grep for only what's relevant, never load the full corpus
  • 4 Skills are lazy-loaded — not included in every prompt, only in the agent that needs it
  • 5 Scratchpad is ephemeral — wipe between tasks; it is working context, not storage
  • 6 Security is a gate — pipeline stops on BLOCKERS, never skip the Security agent
  • 7 Token logs are feedback — review weekly and tune; identify expensive agents
  • 8 Classification is a gate, not a suggestion — if classify_task.sh returns FORCE_FULL, the orchestrator does not override it

Routing Decision Tree (Rule #8 in practice)

classify_task.sh runs (PreToolUse hook)
  │
  ├─ Hard rule matched? ──► FORCE_FULL
  │                              └─ Orchestrator dispatches Full pipeline
  │                                 (no override possible)
  │
  └─ No match ──────────► AMBIGUOUS
                                └─ Orchestrator reasons inline (~200 tokens)
                                       ├─ New behavior / shared logic / risk?
                                       │       └─ YES → Full pipeline (log reason)
                                       └─ Small, isolated, no risk?
                                               └─ NO  → Fast-track (log reason)
